Flight Controls

Embedded controls, state estimation, and hover logic built from first principles.

This project arc began with direct quadrotor motor actuation and raw IMU inspection, then grew into a full estimation-and-control stack: actuator calibration, complementary-filter attitude estimation, control allocation, height and horizontal-velocity estimation, and stable hover tuning.

Project Scope

A complete controls story, not just isolated lab deliverables.

The work evolved from low-level motor commands into a structured flight-control pipeline that combined embedded C++, rigid-body dynamics, sensor fusion, control tuning, and MATLAB-based validation. By the end, the vehicle could estimate its orientation, altitude, and horizontal motion and use those states to stabilize hover in a noisy simulated environment.

The most important part of this project was not any single lab task. It was understanding how modeling, estimation, and control depend on one another in a real flight loop, and then using that understanding to tune a system that actually behaves better in practice.

System Foundations

From raw actuation to physically meaningful motor models

The first step was understanding the quadrotor at the actuator and sensor level. I started by directly driving all four motors from the embedded flight loop and observing how the IMU responded with the motors on, off, and while the vehicle was manually rotated. That gave me a practical feel for sensor noise, axis conventions, and the difference between a stationary reading, vibration-induced noise, and deliberate motion.

I then moved from raw motor commands to a speed-based actuation model. Using measured data, I treated PWM as a control input and identified a linear relationship between commanded PWM and rotor angular speed. From there, thrust modeling followed the expected quadratic relationship with angular velocity, matching the standard propeller-force form T = C_T \rho \Omega^2. That calibration step was important because it connected the software command layer to physically interpretable rotor behavior instead of arbitrary motor numbers.

This stage of the project taught me how to translate between signals, models, and hardware behavior: PWM into rotor speed, rotor speed into thrust, and thrust into vehicle response. It also created the foundation for the later control-allocation and mixer work, because once each propeller command corresponded to meaningful force, the rest of the controller could be expressed in dynamics rather than guesswork.

Attitude Estimation

Using gyroscope integration and accelerometer correction together

Once the low-level actuation and IMU behavior were understood, I built orientation estimation for roll, pitch, and yaw. The gyroscope alone provides angular rate, so attitude can be estimated by integration, but the estimate drifts over time because bias and noise accumulate. To reduce that drift, I used accelerometer-based roll and pitch measurements as a slow corrective reference while preserving the gyroscope's ability to capture fast motion.

In practice, this became a complementary filter. The estimator used the standard form of combining integrated gyro attitude with accelerometer-derived attitude using a mixing factor \rho. A smaller \rho trusts the gyroscope more and corrects slowly; a larger \rho trusts the accelerometer more and responds more like a noisy step measurement. By testing multiple values, I learned how estimator tuning changes the tradeoff between drift rejection and measurement noise sensitivity.

The theory mattered because the estimator is what the controller sees as the vehicle state. If the state is wrong, the controller is reacting to fiction. This stage made the link between estimation and control very tangible: bias calibration, sensor fusion, and discrete-time update logic all directly affected how well the later control loops behaved.

Attitude Control

Closing the loop with cascaded rate logic and control allocation

After building attitude estimation, I implemented closed-loop control for roll, pitch, and yaw. The structure was effectively a cascaded controller: attitude error produced desired body rates, and the measured body rates then fed a faster rate loop that generated commanded angular accelerations. Those desired rotational dynamics were converted into body moments using the vehicle inertias.

The next key step was control allocation through the mixer matrix. Total thrust and the desired moments about the body axes were distributed across the four propellers using the geometry of the quadrotor arm length l and the yaw-coupling coefficient k. This is where the earlier thrust modeling became essential: the controller did not simply produce “motor numbers,” it produced desired forces and torques that were then mapped into motor commands through a physically motivated transformation.

I also evaluated intentionally unstable and low-thrust cases, which was valuable because it exposed how time constants, available authority, and over-aggressive corrections change system behavior. Those tests made the stability margins feel real: if the controller is tuned too aggressively or the available thrust is limited, the resulting pitch response becomes choppy, delayed, or outright unstable.

Hover Logic

Estimating height and planar motion before trying to hold position

Stable hover required more than attitude control. The vehicle also needed usable estimates of altitude, vertical velocity, and horizontal velocity. For altitude, I used the range sensor but corrected the raw height reading by dividing through the attitude-dependent geometry term \cos(\phi)\cos(\theta), since a tilted vehicle sees a different slant distance than a level one. Height and vertical velocity were then blended in a predictor-corrector style update using a measurement mixing factor.

For horizontal motion, I used optical-flow measurements together with gyroscope compensation and the current height estimate. The core idea was that image motion alone is not pure translation; part of it is caused by body rotation. By subtracting out the rotational contribution and scaling by the estimated height, I could turn optical-flow readings into horizontal velocity estimates in the vehicle frame.

With those states in hand, the controller added two more layers: horizontal damping and vertical height control. Horizontal velocity was driven toward zero using a time-constant-based acceleration command that then mapped into desired roll and pitch. Altitude control used a second-order target with natural frequency and damping ratio, turning height error and vertical velocity into a desired vertical acceleration. That desired acceleration fed back into the same thrust-and-moment framework from the attitude controller.

In other words, the final hover logic was not a special-case script. It was the result of layering translation estimation and outer-loop commands on top of an already working attitude-control architecture.

Validation and Tuning

Using telemetry and iterative tuning to extend stable flight

The final phase of the project was tuning and validation. I used MATLAB to analyze logged telemetry for desired and estimated angles, height, velocities, and total motor force, then iteratively adjusted time constants and estimator mixing values. This was especially important in hover, where a mathematically valid controller can still oscillate badly if the gains are too reactive or the state estimates are too noisy.

The tuning process was closer to engineering practice than to a purely symbolic exercise. Small changes in the rate and angle time constants altered how aggressively the vehicle corrected attitude. The mixing parameters for height and horizontal velocity changed how much the controller trusted its measured states. In the final hover tests, those adjustments noticeably improved flight duration and reduced the aggressive overcorrection that had previously driven the vehicle into the walls.

The biggest lesson from this project was that a flight-control system only works when modeling, estimation, actuation, and tuning all agree with one another. A controller is only as good as the states it receives, and state estimates are only useful if they are interpreted through the actual vehicle geometry and sensor physics.
